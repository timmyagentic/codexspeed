# GPT-5.6 Series Run Design

## Goal

Add an explicit, reproducible GPT-5.6 series scope to CodexSpeed, release it as
Runner v0.1.3, deploy the compatible website, then execute and publish one
standard GPT-5.6 series benchmark from this Mac.

The approved benchmark contains GPT-5.6-Sol, GPT-5.6-Terra, and GPT-5.6-Luna at
every comparable effort exposed by the live catalog. Ultra remains excluded by
the existing single-agent methodology.

## Scope semantics

The CLI accepts one new option:

```text
--series gpt-5.6
```

A series identifier uses the same bounded identifier character set as model
IDs. It matches a visible model when the model ID is exactly the series ID or
starts with `<series>-`. Matching is case-sensitive and the runner refuses a
series with no visible comparable models.

Series mode always means the standard benchmark schedule: one unmeasured
warm-up per selected model and three measured rounds across every comparable
cell. `--series` cannot be combined with `--model`, `--effort`, a non-three
round count, or `--no-warmup`. `--seed` and the mandatory `--max-turns` guard
remain available. Ultra is never scheduled.

For the current catalog, `--series gpt-5.6` selects 15 cells:

- GPT-5.6-Sol: low, medium, high, xhigh, max;
- GPT-5.6-Terra: low, medium, high, xhigh, max;
- GPT-5.6-Luna: low, medium, high, xhigh, max.

The exact schedule is three warm-ups plus 45 measured turns, or 48 sequential
turns total.

## Public contract and validation

The run mode enum adds `series`. `selection.series` is an optional bounded
identifier and is present only for series runs. Cross-field validation requires
all of the following for a series artifact:

- `mode` is `series` if and only if `selection.series` is present;
- every selected model belongs to the declared series;
- every visible catalog model in the series contributes all of its comparable
  efforts;
- no hidden, unsupported, or Ultra cell is invented or omitted;
- the standard warm-up and measured-round counts are recorded.

Existing schema-version-1 smoke and full artifacts remain valid without a
series field. The D1 schema does not change because the immutable public JSON
already stores the selection. Run-list queries project the optional series
identifier from that JSON and return `null` for legacy runs.

## Runner and data flow

`plan --series gpt-5.6 --seed 13 --max-turns 48` discovers the live App Server
catalog and prints the exact bounded schedule without starting a model turn.
The matching `run` command uses the same options, isolated temporary
`CODEX_HOME`, fixed prompt, default service tier, sequential execution, and
privacy allow-list as every existing benchmark.

The artifact records `mode: "series"`, `selection.series: "gpt-5.6"`, Runner
v0.1.3, the catalog snapshot, all 48 samples, and the existing timing/token
evidence. It never records prompt, response, reasoning, credentials, local
paths, transcripts, or arbitrary environment data.

## Website and API presentation

All public surfaces use one scope-label formatter:

- smoke -> `Smoke run`;
- full -> `Full run`;
- series `gpt-5.6` -> `GPT-5.6 Series run`.

The label appears in the latest-run facts, run history, and run detail heading.
The API returns the series identifier in full artifacts and list metadata while
continuing to parse the existing production smoke runs. The methodology page
explains that a series run covers every comparable cell in one named series but
is not a full-catalog run.

## Failure and publication behavior

There are no silent retries. A failed or invalid model turn remains in the
completed immutable artifact and contributes to reliability counts. After the
run, the artifact is checked for schema validity, exact 15-cell coverage, three
warm-ups, 45 measured samples, absence of Ultra, owner-only permissions, and
the sensitive-field deny-list before publication.

The existing Keychain publisher secret signs the exact artifact bytes. The
first upload must return `201 created`, a byte-identical repeat must return
`200 already published`, and production detail/latest/list responses must match
the local artifact and recomputed summary.

## Release and verification

Implementation follows strict test-driven development. Focused contract,
scheduler, CLI, Worker, UI, and browser tests are added before production code.
The full repository gate and GitHub CI must pass before merge.

After merge, create and verify immutable GitHub release `v0.1.3`, deploy the
compatible Worker/static assets, and build the runner from a detached checkout
of that exact tag. Re-run `doctor` and the no-turn 48-turn plan from the tag
before starting the model run. The production handoff includes API/D1/cache/
security checks and desktop plus 375-pixel browser, accessibility, console, and
network verification.
